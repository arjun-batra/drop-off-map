### The orchestrator's critical flag: `/api/geocode` behind `AuthGate.check()` — independently verified, PASS

This was treated as the single most important thing to verify in this increment, per the orchestrator's explicit instruction and reviewer's carried-forward INC-1 note. Verified at **two independent levels**, not just by reading dev's handoff claim:

1. **Direct handler invocation** (`tests/api/geocode.test.ts`), covering the full matrix: `free_tier` (no cookie needed, request reaches the provider); `paid_tier` + no cookie → `401 unauthorized`; `paid_tier` + wrong/garbage cookie → `401`; `paid_tier` + a cookie valid under a since-rotated password → `401`; `paid_tier` + a valid session cookie → passes through to the provider call and returns `200`. Critically, for every `401` case, the test asserts the provider `fetch` spy was **never called** — this proves `AuthGate.check()` runs and short-circuits *before* any billable provider call is attempted, not merely that the final HTTP status happens to be 401 for some unrelated reason (e.g., a coincidental downstream error). Also confirmed `AuthGate.check()` is evaluated before the `query`-shape validation (a too-short query with no auth still returns `401`, not `400` — the wrong body would leak information about internal validation rules to an unauthenticated caller).
2. **Real HTTP, against the actual local dev server** (`npm run dev`, hitting `localhost:5173` with `curl`, exactly the way a real client would) — independently reproducing dev's own claimed smoke test rather than trusting it: with `APP_MODE=paid_tier`, `GET /api/geocode?query=...` with no cookie returned `401 {"error":"unauthorized"}`; with a garbage cookie, also `401`; after `POST /api/auth/verify-password` with the correct password and reusing the resulting `Set-Cookie` value, the same request passed AuthGate and reached the (deliberately invalid, dummy) Google API key, correctly surfacing `502 {"error":"provider_error","message":"The provided API key is invalid. "}` — i.e., the request provably passed through `AuthGate` and all the way to the real outbound provider call, not blocked or short-circuited anywhere in between. Also confirmed `free_tier` mode reaches the provider with **no cookie at all** (same `502`, not `401`), confirming the gate is mode-sensitive in both directions, not just "always blocks" or "always passes." `GET /api/config/public` was reconfirmed to remain `200`/unauthenticated in `paid_tier` mode throughout (INC-1 behavior, unaffected by this increment).

**Conclusion: dev's claim is correct.** `/api/geocode` is genuinely, provably wrapped by `AuthGate.check()`, checked before any provider call, in both directions (blocks correctly in `paid_tier`, does not block in `free_tier`). This closes reviewer's REV-002-adjacent carry-forward note from the INC-1 audit ("AuthGate's real test comes when the first protected business endpoint lands") with a PASS.

**BUG-001 [FUNCTIONAL] — `SearchFlow.tsx`'s Loading→Results/Error transition has no request-cancellation guard; clicking "Cancel" does not actually cancel the in-flight search, and a stale response can silently overwrite the user's current screen/data.**

- **Increment**: INC-6.
- **File**: `src/frontend/components/SearchFlow.tsx`, `runSearch()`.
- **FR/NFR relevance**: Not a direct violation of FR-006f/FR-010/FR-011/FR-012/FR-013 (the ranking/fallback/no-viable/display logic itself is correct once a response is rendered) — this is a UI state-management defect in this increment's own new orchestration code (`SearchFlow.tsx`, built this increment to satisfy design.md section 8's "frontend renders off `.status`" contract and ux-spec.md's Input→Loading→Results/Error flow, sections 5-7). It directly contradicts ux-spec.md section 5's documented Cancel behavior ("returns the user to the Input screen") by not making that return durable, and risks showing the user incorrect/stale search results without any action on their part, which is a real trust/correctness problem for a results-display feature. Treating it as a genuine bug against this increment's own claimed behavior, not merely a nice-to-have.
- **Reproduction steps** (verified two independent ways):
  1. **Real browser** (Playwright, mobile viewport): fill all 4 fields, click "Find drop-off points" (search intercepted with a 2-second artificial delay), wait for the Loading screen, click "Cancel" (returns to Input screen immediately, as expected), then wait past the 2-second delay. **Actual**: the app silently jumps from the Input screen to the Results screen showing the (stale, already-cancelled) search's results, with no user action. **Expected** (ux-spec.md section 5): the user stays on the Input screen after cancelling; a request they explicitly cancelled should never later hijack their screen.
  2. **Automated regression** (`tests/frontend/SearchFlow.test.tsx`, `it.fails(...)`, deterministic via a manually-resolved promise instead of a timer): same sequence — submit, cancel while the promise is still pending, resolve the stale promise afterward, assert the user is still on the Input screen and the stale candidate label is not shown. This assertion currently **fails** (confirming the bug), which is why the test is marked `it.fails` rather than a normal `it(...)` — flip it back to a normal assertion once this is fixed, at which point it should pass and continue guarding against regression.
- **Root cause**: `runSearch()` calls `setStage({ kind: "loading", ... })`, awaits `searchDropOffPoints(request)`, and then unconditionally calls `setStage(...)` again with the result — there is no `AbortController`, request-id/sequence-number check, or any guard confirming the stage the app is *currently* in still corresponds to the request that just resolved. `onCancel` only sets `stage` back to `"input"`; it does not cancel or invalidate the in-flight promise chain.
- **Compounding risk beyond the reproduced scenario (not separately tested, but follows from the same root cause, flagging for dev's fix design)**: if a user cancels Request A and then immediately submits a new Request B that resolves *faster* than A, a slow Request A resolving afterward would overwrite B's fresh results with A's stale ones — i.e., this isn't limited to "stale results appear after Cancel," it can also silently replace a **newer, correct** result with an **older, wrong** one. Recommend the fix (e.g., an `AbortController` per request, or a monotonically increasing request-sequence number checked before each `setStage` call in `runSearch`) address both manifestations, not just the Cancel case specifically.
- **Expected vs. actual**: Expected — once the user cancels, the app stays on the Input screen regardless of what a stale in-flight request later returns. Actual — the stale request's resolution silently overwrites the current screen.
- **Owner**: dev.
