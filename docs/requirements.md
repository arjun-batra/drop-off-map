# Requirements: Drop-off Point Optimizer

Status: FINAL — all open questions resolved, presented for Gate 2 approval.
Source: docs/idea-brief.md (approved, Gate 1 passed 2026-07-10).

## 1. Functional Requirements

### Inputs
- **FR-001**: The system shall accept three location inputs per session: driver's start point, driver's destination, and the family member's (passenger's) destination.
- **FR-002**: The system shall accept a user-supplied maximum acceptable added-detour time, in minutes, entered per session (not a fixed default).
- **FR-003**: The system shall validate that all three input locations resolve to a valid geographic point before computing results, and shall display a clear error message if any location cannot be resolved.
- **FR-004**: If the start or destination point falls outside the configured geographic service radius (see Configuration: `GEOGRAPHIC_RADIUS_KM`), the system shall block the request and display a message stating that the app currently only serves within that radius. The system shall not attempt a best-effort calculation outside the service area.

### Computation
- **FR-005**: The system shall identify candidate drop-off points along the driver's route between start and destination.
- **FR-006**: For each candidate drop-off point, the system shall compute:
  a. Driving time from the driver's start point to the drop-off point (using live traffic conditions),
  b. Added detour time, defined as: (drive time from start, via the drop-off point, to the driver's destination) minus (direct drive time from start straight to the driver's destination, no stop),
  c. Walking time from the drop-off point to the nearest usable transit stop,
  d. Transit time from that stop to the passenger's destination (including any transfers), using live transit departure data,
  e. Total time for the passenger from drop-off to destination (walk + wait + transit + any transfer time),
  f. Total time for the driver from start to their own destination via the drop-off point.
- **FR-007**: The system shall use live driving traffic conditions (not static/historical-average travel times) when computing driving and detour times.
- **FR-008**: The system shall use live transit departure/schedule data (not static timetable-only data) when computing walking, wait, and transit times.
- **FR-009**: The system shall filter candidates to those whose added detour time (FR-006b) is less than or equal to the user-supplied threshold (FR-002).
- **FR-010**: Among qualifying candidates (FR-009), the system shall rank and present the top N candidates (default N=3, see Configuration: `MAX_CANDIDATES_RETURNED`), ordered by ascending passenger total time (FR-006e). Where two or more candidates have equal passenger total time, the candidate that occurs earlier along the driver's route shall be ranked higher.
- **FR-011**: If zero candidates satisfy the detour threshold (FR-009), the system shall present the single candidate with the smallest passenger total time (FR-006e) — regardless of its detour time — along with a visible warning that it exceeds the requested detour threshold.
- **FR-012**: The system shall display a clear message (not a silent empty result) if no viable drop-off/transit combination exists at all (e.g., no transit service reachable from any point along the route). Clarification (see DEC-3): a candidate whose best route to the passenger's destination is walking-only (no transit legs) is a viable result, not a no-viable-option trigger — it is ranked normally per FR-006/FR-010 with a near-zero transit time. The "no viable option" condition in this FR applies only when a candidate has neither a usable transit itinerary nor a viable walking-only path.

### Output / Display
- **FR-013**: For each returned candidate, the system shall display the full time breakdown from FR-006 (drive time, detour time, walk time, transit time, total passenger time, total driver time).
- **FR-014**: The system shall display a persistent, visible disclaimer on all results stating that the suggested drop-off point is an estimate and that the driver is responsible for confirming it is a safe and legal place to stop.
- **FR-015**: The system shall allow the user to input each of the three required locations either by typing an address (validated/resolved via a geocoding or places lookup) or by selecting "use my current location."

### Access / Cost Control
- **FR-016**: The system shall operate in one of two exclusive app-wide modes: "free-tier mode" (fully open to the public, no password required) or "paid-tier mode" (the entire app, including all functionality, requires a correct password before any use). There is no partial/feature-specific gating.
- **FR-017**: The system shall provide a configurable mechanism (see Configuration: `APP_MODE`, `PAID_TIER_ACCESS_PASSWORD`) for switching the app between free-tier mode and paid-tier mode. The switch is an operator-controlled configuration setting; the system is not required to detect usage thresholds and switch modes automatically in v1.

### Toll Avoidance & Highway Exclusion — Change Request 2026-07-12 (STATUS: FR-018–FR-022 FINAL, APPROVED)

FR-018 through FR-022 are new/changed requirements arising from a post-launch change request. FR-018 and FR-019 were drafted pending two blocking open questions (OQ-9, OQ-10); both have now been answered by the user and are reflected below. FR-020 and FR-021 were approved as originally drafted, with no changes. FR-022 (Google Maps rendering) was added following separate user approval of change-request item 4. See Changelog entries dated 2026-07-12 for the full history.

- **FR-018**: The system shall provide an "avoid tolls" checkbox as a per-session input control (default unchecked/tolls-allowed, no persistence across sessions per NFR-003). When checked, every driving-route calculation performed for the driver — the direct baseline route (FR-006b's "no stop" baseline), the start-to-candidate leg, and the candidate-to-destination leg, for every candidate evaluated — shall exclude toll roads entirely. This is a hard exclusion, not a preference, cost penalty, or soft ranking factor: a driver-side route that uses any toll road must never be returned or offered when this checkbox is checked. This requirement applies only to the driver's route calculations; it has no effect on the passenger's transit itinerary calculation (FR-006c/d). **(OQ-9 resolved)** If, for a given start/destination pair, no toll-free driving route exists at all (i.e., every possible route includes a toll road), the system shall display the message "No toll-free route found." This is a message-only/empty-result outcome — similar in shape to the existing FR-012 "no viable option" pattern — not a fallback to a toll-inclusive route and not a warning-plus-result display.

- **FR-019**: When the "avoid tolls" checkbox (FR-018) is unchecked (tolls allowed) and the driver's calculated route for a given candidate includes a toll-road segment that the route exits and later re-enters within the same trip, the system shall present the user with a question asking whether this exit/re-entry pattern is acceptable. This question shall only be presented when both conditions hold: (a) FR-018's checkbox is unchecked, and (b) the calculated route actually contains a toll-road exit followed by a later re-entry of a toll road within the same trip. When FR-018's checkbox is checked (tolls avoided entirely), this question shall never be presented, since no toll segment exists to exit or re-enter. **(OQ-10 resolved)** If the user answers that the exit/re-entry pattern is not acceptable, the system shall exclude that specific candidate/route from the results entirely. This is a straightforward exclusion: the system shall not attempt a targeted recalculation to route around the re-entry pattern, and shall not merely record/display the concern while retaining the candidate — the candidate is simply removed from the candidate list, subject to normal FR-010/FR-011/FR-012 handling of the remaining (or now-zero) candidates.

- **FR-020**: The system shall never present a candidate drop-off point that falls on a limited-access, controlled-access highway (e.g., Ontario's 400-series highways, the QEW, the DVP) — this is a hard rejection, not a ranking penalty; such points shall be excluded from the candidate list before ranking (FR-010) and shall not appear via the FR-011 fallback either. "Highway" for purposes of this exclusion means a functionally limited-access freeway (grade-separated, no direct property access, entry/exit via ramps only) — it explicitly does NOT include arterial/surface roads that happen to carry the word "Highway" in their official name but function as ordinary streets with at-grade intersections, traffic signals, and direct property access. If every raw candidate along a route is excluded under this rule (leaving zero viable candidates), this falls under the existing FR-012 "no viable option" handling — no new fallback behavior is introduced by this FR.

- **FR-021**: For every candidate drop-off point displayed to the user (not only the top-ranked/first candidate), the system shall display, for BOTH the drop-off transit stop (where the passenger boards) and the passenger's destination arrival stop (where the passenger disembarks):
  a. the transit stop's location/name as returned by the transit data provider,
  b. the transit line/route name or number,
  c. the direction of travel (e.g., destination-bound headsign or equivalent identifier).
  This applies uniformly to every candidate shown in the results (top pick and all others), not solely the first/highest-ranked one. For a walking-only candidate (per DEC-3, where no transit leg exists), items (b) and (c) do not apply since no transit line exists to display; only walking-route information is shown for that candidate, consistent with DEC-3's existing treatment.

- **FR-022**: The Results-screen map shall render using Google's Maps JavaScript API (Dynamic Maps), replacing the Leaflet + non-Google tile provider implementation delivered at INC-9. This reverses the decision recorded in design.md §3.1a, which selected Leaflet with a non-Google tile provider specifically to avoid a shared-credit cost risk with Google's routing/transit API usage. That basis no longer holds: Google's pricing model has changed such that Maps JavaScript API (Dynamic Maps) now carries its own separate ~10,000 free-loads/month allotment, distinct from the routing/transit APIs' quota, making the switch low-to-zero added cost at expected usage rather than a shared-credit risk. Tech-lead is separately updating design.md's cost model to reflect this. No change to underlying candidate computation, ranking, or the data displayed on the map (FR-005 through FR-013) is implied by this FR — it is a rendering-technology change only.

No new Configuration entries are introduced by FR-018 through FR-021 — the toll checkbox and re-entry question are per-session user inputs (like FR-002's detour threshold), not operator-controlled tunables, and the highway/transit-display rules are fixed business rules, not user-configurable values. FR-022 does introduce a new Configuration entry: `GOOGLE_MAPS_JS_API_KEY`, a distinct key from `MAP_API_KEY`, per the user's decision at DEC-7 (§5) approving tech-lead's recommendation. See the Configuration table below.

**Item 5 (UI modernization)** is a full UX audit/redesign mandate (referencing Google Maps, Waze, and Apple Maps as aesthetic/UX benchmarks), not a new functional requirement — no FR is drafted for it. It is routed to designer to produce/update `docs/ux-spec.md` accordingly.

## 2. Non-Functional Requirements

- **NFR-001**: The web app shall be responsive and usable on mobile browser viewports without requiring installation of a native app.
- **NFR-002**: The system shall not require user accounts, login, or authentication for baseline (free-tier) use.
- **NFR-003**: The system shall not persist user-submitted location data beyond what is required to serve a single request; no trip history, favorites, or cross-session storage shall be implemented in v1.
- **NFR-004**: The system shall return results within 5 seconds of the user submitting valid inputs, under normal operating conditions. *Note: this target depends on the latency of the chosen third-party routing/transit data provider(s) and the number of candidate points that must be evaluated per request. Tech-lead must validate feasibility of this target during design (Phase 2); if 5 seconds is not achievable given the chosen data provider(s) and candidate-evaluation approach, this must be escalated back to pm/user as a design-stage trade-off (e.g., fewer candidates evaluated, provider change, or a revised target) rather than silently missed.*
- **NFR-005**: The system shall be deployed to a live, publicly reachable URL, with CI/CD and hosting established before the first increment begins (per delivery workflow).
- **NFR-006**: The system's geographic service area shall be limited to a configurable radius (default 200km) around a configurable center point (default Toronto).
- **NFR-007**: Rate-limiting/abuse protection on free-tier (ungated) usage is explicitly out of scope for v1/MVP (deferred). This is a known, accepted risk: a single user or bot could exhaust free-tier API quota and force the app into paid-tier mode (FR-016) sooner than expected. No action is required beyond the operator manually switching `APP_MODE` if this occurs.

Decided: walking-only routes are treated as a valid result with near-zero transit time and ranked normally alongside all other candidates per FR-006e/FR-010 (likely near the top, since a short walk to the destination is a great outcome for the passenger) — they are not flagged as no-transit-available and do not trigger the FR-011 fallback or the FR-012 "no viable option" message. FR-012 updated with a clarifying note reflecting this.
- **DEC-4 (2026-07-12) — Lower free-tier submission ceiling accepted as the new reality; no mitigation requested (relates to DQ-2 in `docs/design.md` §1.2, FR-016/FR-017's `APP_MODE` mechanism, NFR-007, and Configuration entry `MAX_TRANSIT_EVALUATIONS_PER_REQUEST`; originates from tech-lead's 2026-07-12 cost-model refresh, `docs/design.md` §3).** What was found: tech-lead discovered that Google Maps Platform eliminated its shared ~$200/month pooled usage credit (effective 2025-03-01), replacing it with per-SKU monthly free-call allotments. Under the new per-SKU model, the Directions API (Legacy) SKU — shared between the single driving-route call and up to `MAX_TRANSIT_EVALUATIONS_PER_REQUEST` (default 8) transit-mode evaluation calls per submission — is the binding constraint, recomputing the real free-tier ceiling to roughly ~550-600 submissions/month. This is a sharp reduction from the ~2,000-2,500/month figure the user originally approved at DQ-2 (2026-07-11), because that original estimate was built on the now-eliminated shared-credit model, not because the app's per-submission dollar cost changed. Decided: the user was informed of the revised ~550-600/month figure and its cause, and explicitly said "Yes, accept lower free-tier ceiling as is." No mitigation was requested — specifically, no change to `MAX_TRANSIT_EVALUATIONS_PER_REQUEST` or any other configuration value was asked for; the user accepts the lower ceiling as the new operating reality for `APP_MODE=free_tier` (FR-016/FR-017). This decision supersedes/updates only the *numeric assumption* underlying DQ-2's resolution in `docs/design.md` §1.2 — DQ-2 itself remains historically accurate as "the estimate the user approved at the time, based on the pricing model that existed at the time"; it is the external pricing model that changed, not the validity of the original approval. No FR/NFR text changes as a result of this decision — this is a record of an accepted-risk decision, not a new requirement.
- **DEC-5 (2026-07-12) — FR-018 toll avoidance: "best-effort with partial verification" accepted as satisfying FR-018's "must never be returned" language; no upgrade to Routes API's paid toll add-on requested (relates to FR-018; originates from tech-lead's `docs/design.md` §1.4 flag, first bullet).** Considered: tech-lead flagged that Google's `avoid=tolls` parameter is documented by Google as a preference, not a hard constraint — a toll route may still be returned if no reasonable toll-free alternative exists — and that neither Directions nor Distance Matrix returns a structured per-route "contains a toll" flag for after-the-fact verification. Distance Matrix, used to batch-evaluate most of the raw candidate pool, returns no step/road-name data at all, so toll-freeness can only be verified (via text-pattern-matching against a known regional toll-road identifier list, itself an approximation, per §4.7) for the direct-baseline route and the small set of final candidates that reach step-level Directions data — not for every raw candidate generated. The alternative would be migrating driving-route calls to Google's newer Routes API and paying for its `extraComputations: ["TOLLS"]` add-on for a stronger, verified guarantee across the full candidate pool, at materially different cost. Decided: the user explicitly accepted the best-effort/partial-verification approach as-is — "Best effort with partial verification acceptable." No stronger guarantee (Routes API toll add-on) was requested; no scope or cost trade-off is being pursued for this. This resolves the open question tech-lead recorded in `docs/design.md` §1.4 (FR-018 bullet); tech-lead's own log there no longer needs to track it as open.
- **DEC-6 (2026-07-12) — FR-020 highway exclusion: allowlist-based text-pattern-match against known limited-access highway identifiers accepted as the detection method, with residual false-positive/false-negative risk acknowledged (relates to FR-020; originates from tech-lead's `docs/design.md` §1.4 flag, third bullet, and §4.7's technical assessment).** Considered: tech-lead flagged that neither Directions nor Geocoding exposes any road-classification/limited-access-freeway data field, so there is no way to geometrically or topologically confirm a candidate sits on a true limited-access highway. The design instead applies a curated, Toronto-region-specific allowlist (400-series highways, QEW, DVP, Gardiner Expressway) as a text-pattern-match against the step text already fetched from the direct-route Directions response — an approximation deliberately biased toward precision (avoiding wrongly excluding a valid arterial candidate that happens to carry the word "Highway" in its name, per FR-020's own text), but one that carries real residual risk: a genuine limited-access highway not on the list could be missed (false negative), or a name collision could cause a wrongful exclusion (false positive). Decided: the user explicitly accepted this approximation as-is — "yes" (acceptable as an approximation). No request was made for a stronger/geometric detection method or an expanded/alternate identifier source. This resolves the open question tech-lead recorded in `docs/design.md` §1.4 (FR-020 bullet); tech-lead's own log there no longer needs to track it as open.
- **DEC-7 (2026-07-12) — FR-022 Google Maps JS API key: separate, new `GOOGLE_MAPS_JS_API_KEY` config key approved, distinct from the existing server-side `MAP_API_KEY` (relates to FR-022 and the Configuration section; originates from tech-lead's `docs/design.md` §1.4 flag, fifth bullet, and §7.2's recommendation).** Considered: tech-lead recommended against reusing the existing `MAP_API_KEY` for FR-022's client-side Maps JavaScript API rendering, reasoning that a browser-exposed key needs materially different Google Cloud restrictions (HTTP-referrer-based, Maps-JS-API-scoped only) than the server-side key (which is never sent to the browser and is scoped to Geocoding/Directions/Distance Matrix). Reusing one key for both purposes would mean a leaked/scraped browser-exposed key could be used to call the server-side APIs directly, bypassing every cost control this app relies on (radius gating, `APP_MODE`, candidate-count limits, etc.). Decided: the user explicitly approved tech-lead's recommended approach — "aligned with your recommendation." A new, distinct `GOOGLE_MAPS_JS_API_KEY` config key is approved, added to the Configuration table below (§3). This resolves the open question tech-lead recorded in `docs/design.md` §1.4 (FR-022 bullet); tech-lead's own log there no longer needs to track it as open.
