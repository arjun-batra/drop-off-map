### Pass 5 — Security audit

- **Committed secrets**: `.env.local` exists on disk with a dummy `MAP_API_KEY` and a placeholder `PAID_TIER_ACCESS_PASSWORD` (`correct-horse-battery-staple`), but confirmed **not tracked by git** (absent from `.git/index`; `.gitignore` excludes `.env.local`/`.env`). `.env.example` contains only placeholder text, no real values. **No committed secrets found.**
- **Password comparison**: `src/auth/verifyPassword.ts` hashes both sides to a fixed-length SHA-256 digest before `timingSafeEqual` — correctly constant-time, satisfies design.md §5.2.
- **REV-002 [SECURITY] major** — Session token (`src/auth/session.ts`) is a **deterministic, non-expiring** HMAC-SHA256 of a fixed constant message, keyed only by the shared `PAID_TIER_ACCESS_PASSWORD`. Consequences, independently verified by reading the code (not just taking handoff.md's self-flagged note at face value):
  - The valid cookie value is identical for every user/login and never changes unless the operator rotates the shared password. A copied/leaked cookie value (e.g., shared/public computer, browser history sync, any future XSS, log exposure) grants **indefinite** access, not bounded by any session TTL — the only revocation lever is rotating the password for *all* legitimate users too.
  - This contradicts design.md §5.2's own language: "sets a **short-lived** session cookie on success." As implemented, the cookie is not short-lived — it is a permanent credential-equivalent bounded only by browser-close (no `Max-Age`/`Expires` at all) and manual password rotation. This is a design-doc/implementation mismatch, not just a hardening nice-to-have.
  - Cookie flags themselves are otherwise sound: `HttpOnly` (prevents JS/XSS read), `Secure` correctly gated on `VERCEL_ENV` (production/preview only, appropriately omitted for local plain-HTTP dev), `SameSite=Lax` (reasonable; `Strict` would be marginally tighter but Lax is an acceptable default given no other authenticated mutating endpoints exist yet).
  - **Recommendation** (for tech-lead to confirm/design, dev to implement): embed a signed expiry into the token itself (e.g., HMAC over `expiryTimestamp + fixed message`, cookie value `expiryTimestamp.hmac`, `isValidSessionToken` checks both the HMAC and `now < expiryTimestamp`) — this preserves the stateless/no-DB architecture (NFR-003) while actually satisfying "short-lived." Needs a config key added to design.md §7 (e.g., `SESSION_LIFETIME_SECONDS`) if a tunable is wanted, per the project's no-hardcoded-tunables rule.
  - **Severity/routing**: Major, not a blocker for INC-2 (no protected business endpoint exists yet, so today's actual exposure is limited to the password gate's own screen). Must be resolved by INC-8 ("paid-tier hardening... re-auth behavior" is already that increment's explicit scope per design.md §10) or sooner if `APP_MODE=paid_tier` is turned on in a real production deployment before then. **Owner: tech-lead** (amend design.md §5.2/§7 to specify the intended session lifetime and mechanism), then **dev** to implement.
- **REV-003 [SECURITY] minor/informational** — `POST /api/auth/verify-password` has no rate-limiting, attempt counter, or lockout (confirmed in code: no such logic exists in `api/auth/verify-password.ts`). This is an explicit, already-documented user decision (ux-spec.md §3: "Resolved: no lockout/throttling is required for the password gate"), so it is not a defect — but I want to flag independently that it does carry a real consequence design.md itself names as the point of `APP_MODE=paid_tier`: "the password gate itself is also a natural cost-control lever" (design.md §3). An unthrottled, unlimited-attempt online password gate is brute-forceable at whatever rate the network/serverless platform allows, which weakens that stated cost-control rationale if the configured password is short or guessable. Not filing as a blocker (explicit user decision already on record), but recommending **pm** confirm with the user that this trade-off is still acceptable now that it's framed against the gate's stated cost-control purpose, not just its UX purpose. **Owner: pm.**
- **`GET /api/config/public` unauthenticated in `paid_tier` — independent conclusion: CONCUR with dev/QA, acceptable, not a bug.** Verified myself (not just trusting the existing tests) that `toPublicConfig()` only ever returns `{ appMode, geographicCenter, geographicRadiusKm, maxCandidatesReturned, transitModesIncluded }` and that `mapApiKey`/`paidTierAccessPassword` are structurally absent from `AppConfig → PublicConfig` mapping (not just "usually" excluded — there is no code path that could include them). Reasons this is correct, independent of dev/QA's framing:
  1. It is a genuine chicken-and-egg requirement: the frontend cannot decide whether to render the password gate at all without first knowing `appMode`, so *some* pre-auth endpoint must exist.
  2. runbook.md §6 depends on exactly this endpoint, exactly this shape, as its zero-cost/no-auth deploy health check.
  3. FR-016 gates "functionality... any use," not configuration metadata; radius/candidate-count/transit-mode-list are not user data or billable business features.
  One residual, low-severity observation not raised by dev/QA: this endpoint also returns the config-loader's raw validation-error message text (`api/config/public.ts`'s catch branch) to an unauthenticated caller on a 500. This reveals internal config-key names/validation rules to anyone, even in `paid_tier` mode. Low value to an attacker (the same key names are already documented in this repo's own runbook.md/design.md), so **not logged as a standalone finding**, just noted for awareness.
- **`APP_MODE=paid_tier` client-render + server-cookie gate, not edge/CDN-level — independent conclusion: CONCUR, acceptable for INC-1's actual scope, given:**
  1. Zero protected business endpoints exist yet in this increment (confirmed: `api/` contains only `config/public.ts`, intentionally public, and `auth/verify-password.ts`, the gate's own endpoint) — so there is nothing of value reachable pre-auth today regardless of where the block happens.
  2. ux-spec.md §3 explicitly assigns the exact mechanism to dev/tech-lead as an implementation choice, and design.md never calls for an edge-level block anywhere in §2, §5, §8, or the INC-1 scope in §10.
  3. `AuthGate.check()` itself is correctly framework-agnostic and reusable, so INC-2+'s real protected endpoints (`/api/geocode`, `/api/drop-off-search`) just need to wrap themselves with it — verified the interface supports that without modification.
  This conclusion is conditional, not a blanket pass: it depends entirely on every future protected endpoint actually being wrapped in `AuthGate.check()`. I will re-verify this explicitly at the INC-2 audit once `/api/geocode` lands, per the same carry-forward flag QA already recorded.
- No SQL/shell/file-path injection surfaces exist yet (no user input reaches a trust boundary of that kind in INC-1 — the only inputs are `password` string compared via HMAC/hash, and env vars read by the operator). No dependency vulnerabilities beyond the 9 pre-existing advisories QA already noted as not introduced by this increment (not independently re-audited here; flagging that reviewer should re-check `npm audit` output directly at the next pass rather than relying solely on QA's characterization).

Confirmed the divergence is real: design.md §5.1 (as already corrected for the `maxDetourMinutes` parameter) still shows `batchEvaluate(..., config: AppConfig)`, while `src/candidates/detourEvaluator.ts:14` defines `DetourEvaluationConfig = Pick<AppConfig, "distanceMatrixBatchSize">` and uses that narrower type. - Every narrowed type contains exactly the fields its function reads (verified directly against each implementation's body) — no module is silently under- or over-scoped relative to what it actually needs.

Full `AppConfig` as the declared parameter type would be the *weaker* choice here, not the stricter one.
- It measurably improves testability (the explicit, stated rationale in `authGate.ts`'s own comment) — unit tests construct a 1-4-field object instead of a full 17-field `AppConfig`, exactly the kind of leanness this audit's Pass 4 rewards.

### Item 1 — REV-002 closure, independently re-derived from source (the specific ask: don't take QA's word for it)

Read `src/auth/session.ts` in full, not summarized:

- **Signed, verifiable expiry.** `createSessionToken(paidTierAccessPassword, expiresAtEpochSeconds)` returns `${expiresAtEpochSeconds}.${computeHmac(...)}`. `computeHmac` hashes `` `${TOKEN_MESSAGE}:${expiresAtEpochSeconds}` `` — the expiry is genuinely inside the signed payload, not appended alongside an unrelated signature. Confirmed a tampered token (attacker edits the plaintext `expiresAtEpochSeconds` prefix to extend it, keeping the original signature suffix) fails: `isValidSessionToken` recomputes `computeHmac` over the *tampered* expiry value and compares against the *original* signature via `timingSafeEqual` — these will not match unless the attacker also recomputes a valid HMAC, which requires the password. This is the exact tamper-proofing property the task asked me to verify, confirmed directly from the arithmetic, not from a comment claiming it.
- **Genuine rejection of expired tokens, no off-by-one/timezone bug.** `isValidSessionToken` returns `now < expiresAtEpochSeconds * 1000` as its final, strict check (line 79) — `now` (default `Date.now()`, unix ms) and `expiresAtEpochSeconds * 1000` are both plain UTC epoch milliseconds; there is no `Date` string parsing, no local-timezone conversion, no calendar arithmetic anywhere in this function or in `computeHmac`/`createSessionToken` — so no timezone bug is structurally possible (there is nothing timezone-*sensitive* in the computation to get wrong). The comparison is strict `<`, not `<=`, so a token checked at exactly its expiry instant (`now === expiresAtEpochSeconds * 1000`) is correctly rejected, not treated as still-valid for one more instant — this closes the specific "off-by-one at the boundary" risk the task named, confirmed by reading the operator directly, not by trusting QA's boundary test alone (though QA's test, which I also read, exercises exactly this case and gets the same answer). A malformed token (no `.` separator, non-integer expiry segment) fails closed via early `return false`s (lines 66, 71) before any HMAC computation — verified there is no code path that treats a malformed/legacy (pre-REV-002, single-value) token as implicitly valid-forever.
- **HMAC signature still tamper-proof with the new field added.** The signature covers the full `` `${TOKEN_MESSAGE}:${expiresAtEpochSeconds}` `` string, not just `TOKEN_MESSAGE` with the expiry appended unsigned afterward — confirmed by reading `computeHmac`'s single `.update(...)` call, there is exactly one HMAC input string and it includes both components. The constant-time comparison (`timingSafeEqual`, with a length-check short-circuit before it, identical in shape to the pre-existing, already-audited `verifyPassword.ts` pattern) is unchanged in rigor.
- **Cookie `Max-Age` (defense in depth).** `buildSessionCookieHeader` now requires `maxAgeSeconds` and emits `Max-Age=<floored value>` (`cookies.ts` line 56) — confirmed this is additional, not the security boundary itself (the module's own comment says so, and I independently agree: the server-side `isValidSessionToken` check does not trust or read `Max-Age` at all, so a browser that ignored it would still be correctly rejected server-side).
- **Re-auth behavior — no infinite redirect loop, no bypass.** Traced the full loop: `api/drop-off-search.ts`/`api/geocode.ts` return `401` (via `AuthGate.check`, unchanged blocking logic) → `src/frontend/api.ts`'s `searchDropOffPoints`/`callGeocodeEndpoint` map HTTP `401` specifically to `errorCode: "unauthorized"` (not lumped into `"provider_error"`/`"network_error"`) → `SearchFlow.tsx`'s `runSearch` and `useLocationField.ts`'s debounced-query/geolocation callbacks both check `errorCode === "unauthorized"` and call `onSessionExpired()` instead of falling through to a generic error state → `App.tsx`'s `onSessionExpired` clears the client `sessionStorage` flag (`clearClientSessionFlag`) and flips `authenticated: false`, re-rendering `<PasswordGate>`. This is a **one-directional** transition (authenticated → gate) with no automatic re-fetch/re-check that could loop: the only way back to `authenticated: true` is `PasswordGate`'s `onSuccess` callback, which only fires after a real, successful `POST /api/auth/verify-password` — confirmed there is no `useEffect` or timer anywhere in this chain that could re-trigger `onSessionExpired` or re-attempt the original request automatically. **No bypass**: confirmed `AuthGate.check`'s actual blocking logic (the `if (config.appMode === "free_tier") return true` / cookie-presence / `isValidSessionToken` structure) is byte-for-byte unchanged from the pre-REV-002 version except for the added optional `now` parameter (which defaults to `Date.now()` and is never passed a non-default value by either production call site, confirmed via `grep AuthGate.check` across `api/`) — the re-auth UX wiring is purely additive on the frontend and does not touch, weaken, or special-case the server-side gate itself.
- **Config key genuinely required, no hardcoded fallback.** `sessionLifetimeSeconds` is parsed via the same `parsePositiveNumber(..., integerOnly: true)` helper as every other numeric key in `loader.ts` — fails closed (throws `ConfigError`) if unset, no default value anywhere in code. One minor observation, not logged as a standalone finding: this key is required *unconditionally* (even in pure `free_tier` deployments where it's never read), unlike `PAID_TIER_ACCESS_PASSWORD`'s conditional-on-`appMode` requirement — this is a slight inconsistency in requirement-conditionality, but it matches this project's established INC-1 precedent of loading the full schema regardless of which keys a given deployment mode actually consumes (`MAP_API_KEY` and six other INC-1-irrelevant keys were already required unconditionally from INC-1 onward), so not treated as a defect.

**Independent verdict: REV-002 is genuinely, thoroughly resolved.** This is not a partial fix or a "close to expiry" approximation — signed expiry, tamper-proof signature, strict-boundary rejection, no timezone/off-by-one exposure, and a re-auth UX that neither loops nor creates a new bypass path. Formally closing in the Findings Register (done above).

## Post-Closure Incident Audit — 2026-07-12: REV-018 (ESM relative-import resolution)

**Trigger**: after Phase 4 closure was declared complete and the project was deployed, production went down — every deployed `api/*.ts` handler crashed at cold start with `FUNCTION_INVOCATION_FAILED` (`Error [ERR_MODULE_NOT_FOUND]`). This section is a targeted incident audit, not a re-run of the full nine-increment 5-pass audit: it (1) formally logs the finding (REV-018, above), (2) assesses root cause and forms an independent recurrence-prevention recommendation, and (3) honestly re-examines whether reviewer's own prior audits — including the Phase 4 closure audit above — should have caught this before it shipped. Inputs read directly for this pass: `docs/handoff.md`'s two newest sections (INC-9's tail and the "Critical production hotfix" section), `docs/test-report.md`'s "Hotfix Verification" section, `package.json`, `tsconfig.json`, `.github/workflows/ci.yml`.

### 1. What broke, why it was invisible, how it was fixed, how it was verified

**What broke**: `package.json` has `"type": "module"`, so Node's real ESM module loader is what actually executes every `api/*.ts` handler on Vercel at runtime. Node's ESM loader requires explicit file extensions on relative import specifiers (`from "../../src/config/loader.js"`). The codebase — across all nine increments, starting at INC-1 — wrote every relative import extensionless (`from "../../src/config/loader"`). This is syntactically and semantically valid under `tsconfig.json`'s `"moduleResolution": "Bundler"`, so it type-checked (`tsc --noEmit`), built (`vite build`), and passed the entire 433-test Vitest suite, in every increment, in CI, and in this log's own nine 5-pass audits. None of those layers ever executed a compiled handler under Node's actual ESM resolver — the exact mechanism Vercel's serverless runtime uses — so the defect was never exercised anywhere until a real cold start on Vercel hit it.

**Why it was invisible to the entire pipeline, not just to one agent**: this is the load-bearing fact, independently confirmed by re-reading `package.json`/`tsconfig.json`/`.github/workflows/ci.yml` directly rather than taking dev's framing on trust:
- `tsconfig.json:6` — `"moduleResolution": "Bundler"` — resolves extensionless specifiers by design; this is what `tsc --noEmit` (the `typecheck` script) uses.
- `vite.config.ts`/`npm run build` — Vite's own bundler resolution, same story, and only ever bundles the **frontend** (`src/frontend/**`) in any case — it was never going to exercise `api/*.ts` at all.
- `vitest` (`npm test`) — built on Vite, same bundler-style resolution, confirmed by `tsconfig.json`'s single shared `moduleResolution` setting covering `src`, `api`, `scripts`, and `tests` alike (line 19's `include` list).
- `.github/workflows/ci.yml` lines 59-73 — `lint`, `typecheck`, `test`, `build`: exactly the same four bundler-resolution-based commands, run in CI with no additional step. CI could not have caught this either, as configured.
- No agent in this project's history — dev (nine increments of smoke tests), qa (nine increments of test-report.md passes plus a full Phase 4 E2E run), or reviewer (nine 5-pass audits plus the Phase 4 closure audit) — ever ran a compiled handler under Node's real, unbundled ESM loader before this incident. This is a structural blind spot shared identically by every verification layer in the pipeline, not a one-off gap in any single agent's diligence.

**The fix** (commit `c12510a`, per dev's handoff.md section and QA's Hotfix Verification section): explicit `.js` extensions added to every relative (`./`, `../`) import/export specifier in the 22 files transitively reachable from an `api/*.ts` handler (every `api/*.ts` file plus every non-frontend `src/**/*.ts` file). TypeScript's `moduleResolution: "Bundler"` explicitly permits writing `.js` in a `.ts` file's specifier (resolves it to the sibling `.ts` file for type-checking, emits the specifier as-written) — this is the standard, documented pattern for TS-authored ESM projects, not a workaround. Dev traced the full import chain from every handler (not just pattern-matched file names) to confirm both which files needed the fix and which files (`src/auth/cookies.ts`, `session.ts`, `verifyPassword.ts`, `config/schema.ts`, `geo/types.ts`, `geocoding/errors.ts`, `http/fetchWithTimeout.ts`, `routing/errors.ts`, `search/types.ts`, and all of `src/frontend/**`) were correctly left untouched because they either have no relative imports of their own or are never reachable from the Node-ESM-executed backend path.

**How it was verified — dev's methodology**: per the bug's own root cause, dev correctly recognized that "tests still pass" (Vitest/`tsc`/`vite build`) is necessary but not sufficient proof of the fix, since all three are structurally blind to this exact bug class. ### 2. Root cause category and recurrence-prevention recommendation

**Root cause category: this is a test/verification-strategy gap, not a code-quality, traceability, or requirements gap.** No FR/NFR in requirements.md was violated by the pre-fix code's *intended* behavior (every handler's logic was and remains correct); the defect was entirely in how the module graph resolves at the exact runtime Vercel uses, and every verification layer in this pipeline — dev's smoke tests, QA's Vitest suite (433 tests, nine increments), CI's lint/typecheck/test/build gate, and reviewer's nine 5-pass audits plus the Phase 4 closure audit — shared the identical bundler-resolution blind spot. This is categorically different from every other finding in this Register: REV-010's gap is an *environmental* limitation (no real API key/network available in-sandbox, the check is well-defined and waiting on a resource); REV-018's gap is a *methodological* one (the check was never defined at all, for any agent, at any point, because nothing in the pipeline's design ever asked "does this resolve under Node's real loader").

**Independent recurrence-prevention recommendation** (this is the single most important output of this audit — logging the bug without this would just mean it recurs on the next new relative import):

1. **Primary recommendation — add a CI step that exercises real Node ESM resolution against compiled output, mirroring exactly what dev/QA did manually.** Concretely: add a script (e.g. `npm run verify:esm`) that (a) compiles `api/**/*.ts` + `src/**/*.ts` (excluding `src/frontend/**`) with a dedicated `tsconfig` (`module: ESNext`, preserving the `api/...`/`src/...` relative layout) to a scratch/CI-temp `outDir`, then (b) runs `node --input-type=module -e "await import('<path>')"` (or a small Node script that does the same via a loop) against every compiled `api/*.js` handler, failing the job on any `ERR_MODULE_NOT_FOUND` (or any import-time throw). This requires no Vercel credentials, no network access, and no linked project — it is exactly the mechanism dev/QA already proved catches this exact bug class (positive result on fixed code, exact reproduction on the pre-fix negative control), and it is cheap to run on every PR. This should be added as a new step in `.github/workflows/ci.yml`, alongside the existing lint/typecheck/test/build steps, not as a replacement for any of them (each of those catches a different class of defect; none of them catches this one).
2. **Secondary, complementary recommendation — an ESLint rule enforcing explicit extensions on relative imports, so a new relative import can't silently regress this between CI runs of item 1.** `eslint-plugin-import`'s `import/extensions` rule (or the newer `eslint-plugin-import-x` equivalent) can be configured to require `.js` on relative specifiers project-wide, scoped to the backend-reachable tree (`api/**`, `src/**` excluding `src/frontend/**`, which is bundled by Vite and doesn't need this). This is a fast, no-compile, editor-visible signal (catches the mistake at the moment a new import is typed, not just at CI time) and costs nothing at runtime — recommend tech-lead add this to `eslint.config.js`'s existing flat config. This complements, not substitutes for, item 1: the lint rule prevents a new *extensionless* import from being added, but only a real ESM-resolution check (item 1) can also catch other classes of real-loader-incompatible code that a lint rule wouldn't (e.g. a `require()` call, a JSON import missing an assertion, a case-sensitivity mismatch on a case-insensitive dev filesystem).
3. **Explicitly not recommending, as a primary fix, a full `vercel build`-in-CI step** — the sandbox's own inability to run it (no linked project/credentials, per both dev's and QA's own flagged limitation) generalizes to a real concern about adding Vercel-account credentials to CI (a new secrets-management surface, a new external dependency for every PR's gate, and coupling CI's pass/fail to a third-party service's availability) — this should remain **release's job** at the actual deploy step (a real `vercel build`/deploy dry-run before/immediately after merge, exactly as QA's Hotfix Verification section itself recommended), not CI's per-PR job. Item 1 gets CI to the same *correctness* guarantee (real Node ESM resolution) without that cost or coupling; a real `vercel build` remains valuable as a final, fuller-fidelity check (it also exercises Vercel's own build/bundling step, which item 1 does not) but as a deploy-time gate, not a per-PR one.
4. **Process recommendation for this project specifically**: reviewer is adding "cross-check `package.json`'s `type` field against `tsconfig.json`'s `moduleResolution` setting and grep the backend-reachable tree for extensionless relative imports" as a standing item in its own hardcoding/leanness passes going forward (see item 3 below for why this should have already been standard practice), independent of whether tech-lead/dev adopt items 1-2. This costs nothing beyond a couple of read-only checks reviewer already has the tools for.

### 3. Honest self-assessment: does this change the Phase 4 closure verdict, and should reviewer's own audits have caught it?

**Direct answer: yes, this was a genuine miss in reviewer's own audit methodology — not merely an environmental/tooling limitation like REV-010, and I am not going to soften that.**

Re-examining what was actually available to reviewer at every one of the nine increment audits and the Phase 4 closure audit: reviewer's tools are read-only (`Read`/`Grep`/`Glob`), with no ability to execute code — this is a genuine, real constraint, and it is the correct explanation for why REV-010's real-API-key/real-network sub-conditions were correctly deferred to release rather than something reviewer should have "found a way" to fake (as the Phase 4 closure audit itself put it). **But this bug is different in kind.** The defect was fully detectable via the exact same read-only tools reviewer already had, with no code execution required at all: `package.json:5`'s `"type": "module"` and `tsconfig.json:6`'s `"moduleResolution": "Bundler"` were both directly readable, and a grep across `api/**/*.ts`/`src/**/*.ts` for extensionless relative import specifiers (the same pattern QA later used to confirm dev's fix was complete) would have surfaced every one of the 22 affected files as a static-reasoning exercise — package.json declares Node's real ESM loader will execute this code at runtime; that loader requires extensions; the code has none. No execution, no API key, no network access, no Vercel credentials were needed to notice this — only the specific insight to cross-check those two config files against each other and grep for the pattern, which none of the nine increment audits (including three that directly read `tsconfig.json` for unrelated reasons — REV-005's `include`-list check, the INC-1 audit's initial file inventory, and this incident's own re-read just now) ever did.

**This means the Phase 4 closure audit's "zero open blockers" verdict, while an accurate statement of what was actually checked, was incomplete as a statement of the codebase's true production-readiness** — a genuine, blocker-severity defect (complete production outage on every API endpoint) existed in the codebase at the time of that audit and was not caught, not because it was undetectable, but because the audit's own checklist never included this specific cross-check. I am not retroactively rewriting the Phase 4 Closure Audit section above (per this log's own established convention of never editing past dated entries, only appending corrections in new dated sections) — but I am recording plainly, here, that it should be read alongside this section as: the "zero blockers" conclusion was true of every check this log's methodology actually performed, and false of the codebase's actual production behavior, and the gap between those two things was closable with tools reviewer already had. Going forward (item 4 above), this specific cross-check is now a standing part of reviewer's own hardcoding/leanness passes, precisely so this exact class of self-assessment doesn't need to be honestly re-made after the next production incident instead of before it.

**Bottom line**: REV-018 is marked RESOLVED above because the fix is real, independently double-verified (dev + QA, including a genuine negative control), and — per the user's direct confirmation — live in production with `GET /api/config/public` returning clean `200`s. But closing the finding is not the same as closing the process gap that let it ship undetected past a "zero blockers/majors" closure gate; that process gap is what section 2's recommendations (especially item 1, and reviewer's own item 4) are meant to close.
